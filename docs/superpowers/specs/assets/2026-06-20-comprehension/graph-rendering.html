How should Features ↔ code & Architecture render?

Same graph — shop-api, with the “Refund a payment” path highlighted — drawn four ways. Mermaid is option A. Pick the visual language (you can mix: one for whole-architecture, one for a single feature).

A

Mermaid flowchart (auto-layout)

Familiar, zero-effort, text-defined. But: static (no zoom/pan/live highlight), layout gets tangled past ~20 nodes, generic look, edges cross. Good as an export, weak as the daily surface.
B

Layered graph — “everything points to the domain core”

RECOMMENDED
Columns = hex layers; arrows point inward to the domain. Interactive (zoom/pan, hover, click-to-highlight the feature path live), deterministic layout, and the picture itself teaches the dependency rule. This is mermaid's engine (dagre/ELK) but ours to style & animate.
C

Force-directed graph

Great for “what clusters with what”. But layout is non-deterministic (moves every render), becomes a hairball past a few dozen nodes, and doesn’t express the layer/dependency direction. Better as an optional lens than the default.
D

Feature “trace ribbon” (single feature)

For one feature, a subway line beats a graph. Each stop = a layer the request passes through; the terminus is the outside-world adapter. Dead-simple to read; pairs with B (B for the whole map, D when you open one feature).
